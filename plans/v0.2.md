# v0.2 — The Journey Era

## Thesis

Praxis v0.2 transforms a curriculum into a world. v0.1 built measurement. v0.2 builds place.

---

## Taxonomy

| Term | Meaning | Scope |
|---|---|---|
| Challenge | Any individual learning activity | Universal |
| Stage | Pedagogical domain | Movement, Search, Editing, etc. |
| Family | Related activities within a stage | Stage-scoped, explicitly ordered |
| Layer | Learning purpose | Tutorial, Training, Trial, Boss |
| Region | UX identity of a stage | Frontend-only, presentation |
| Journey | User progression through curriculum | Computed by Journey Engine |
| Hub | Primary entrypoint | Neovim UI |

A challenge belongs to exactly one stage, exactly one family, and exactly one layer.

---

## Architecture Decisions

- **CLI owns truth. Neovim owns experience.** Go computes. Lua presents. Never inverted.
- **No standalone TUI.** The editor is the arena.
- **`plans/` documents intent.** Not UI mockups, not visual concepts, not narratives.
- **Existing tutorials are structurally sacred.** No removal, validator changes, or replay regressions across v0.2.x. Instruction-line edits are permitted when guided by approved archetypes (see CURRICULUM.md — section 2).

---

## Journey Model

| Step | Context | Experience |
|---|---|---|
| Arrival | Onboarding | Welcome, orientation, first action, first success |
| Practice | Tutorial | Single-concept drills, highly guided |
| Synthesis | Training | Combined skills, family-based, composite evaluation |
| Proof | Trial | Multi-skill pressure, constrained execution |
| Application | Boss | Realistic tasks, meaningful application |

Onboarding is a frontend-only flow, not a layer. It is not curriculum, not persistent, not measured.

---

## Invariants

- 56/56 replay tests must pass throughout v0.2.x.
- The Hub always answers *what should I do next?*
- No infrastructure ships before its consumer.
- No feature may require accounts, servers, cloud state, or external services.
- Attempt-state semantics follow the behavior matrix in CURRICULUM.md (section 2). Completions, Success Rate, Mastery, and Confidence form the remaining measurement model.

### v0.2.4 — Canonicalization (shipped)

**Question:** How do components relate canonically?

**Answer:** `DescriptionFor` is the single source of truth for all challenge data.

**Delivered:**
- `Describe` renamed to `DescriptionFor`, returns `(Description, bool)` struct
- `describe_test.go` rewritten: one subtest per field per challenge (`TestDescriptionForCompleteness`) + unknown ID test (`TestDescriptionForUnknown`)
- Old round-trip/evaluation tests removed (covered by completeness)
- `cmd/praxis/main.go`: `describe` uses `DescriptionFor` + `json.NewEncoder`; `record` validates IDs before saving
- `TestRecordUnknownChallenge` — record exits 1 for unknown IDs
- `scripts/generate_catalog.go`: consumes `DescriptionFor` struct, groups by Layer→Stage (not historical packs)
- CHALLENGES.md regenerated with Layer columns and Training section
- All docs audited (CURRICULUM.md, PROGRESSION.md, REFERENCE.md, VISION.md, TESTING.md, AUDIT.md, README.md)
- v0.1-style tests removed from describe_test.go (replaced by canonical completeness)

**Deferred:** Lua frontend canonical representation consumer
